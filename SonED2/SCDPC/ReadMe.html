    Setup and Basic Usage
      This script set is "destructive". This means that the input game files will be modified during the injection process. If the files become corrupt, or the original data must be restored for any reason, the desired files must be restored manually from another unmodified source before performing the Split again.


      One-Time Setup Steps
      • Navigate to the folder that contains "soniccd.exe", create a folder named "ROMulan". The full path name of this folder must not contain any spaces
      • Unpack all files from this archive into the newly-created "ROMulan" folder
      • Run the included Batch File "split.txt" using ROMulan or SonED2's built-in ROMulan functionality to retrieve the game's data


      Every Time Level Editing Steps
      • Using SonED2's "Open Project" option in the "File" menu, navigate to the "Projects" folder included with this fileset and open any of the included Project Files to edit its corresponding level
      • Make any changes using SonED2 as described in its own documentation
      • Use SonED2's "Save Project" option by selecting it in the "File" menu or pressing F3 to save the changes to their respective files (This is important as the edited data is SonED2-internal; ROMulan expects to find all necessary data in the files listed by the injector script)


      Other Modifications
      • With the proper knowledge of the game, other changes can be made by modifying the two Batch Files and/or the Injector Files contained in the "Inject" folder as described by the ROMulan documentation. For relevant information, see the "Game Notes" section.


      Every Time Finalization Steps
      • Run the included Batch File "inject.txt" using ROMulan or SonED2's built-in ROMulan functionality to inject the changes into the game
      • Run "soniccd.exe" from the folder just above the "inject.txt" script. The process is now complete

    Game Notes
      Separate Level Data, Shared Object Location Data

        Due to the one-program-per-level nature of this game, no two levels share Tile or Palette data. This means that different Acts within the same Zone and Time Period can use their own Tilesets and Palettes for either subtle or major changes. This is used in the original game to better vary level layout by creating different sets of Chunks for each level.

        Object Layout Data is shared between Time Periods within the same Act. This is necessary because all Objects for all Time Periods must exist within the data in order for the game to keep track of which Objects have been interacted with in each Time Period. For this reason, the Objects contain Time Period flags, for defining whether or not they do or don't "exist" (show up) in each of the three Time Periods (Past, Present, Good/Bad Future).



       Data File Size Restrictions

        Because some of the data is contained within a PC executable file, it cannot be relocated.

        No data file for this game can grow larger than its original, as it must be placed in the exact location it was extracted from. If it were to become larger, it would overwrite other vital data and cause serious errors.

        The "Plane Size" of the level should not be modified, and the number of Objects used for the level should not be increased beyond its original value.



       Animated Tiles and Palettes

        Certain Tiles and Palette entries are animated/cycled during gameplay. This behavior is not accounted for in the current script set version; these animations and the data that they use are not available for editing. Any modifications to these Tile or Palette entries will be overridden by these animations during gameplay.

        In this game, the Tile data contains number and/or special character graphics at these positions, which is what will be displayed in SonED2.



       Loops and Crossing Paths

        This game uses a one-path collision system, meaning that Chunks within the level layout must be actively changed during gameplay to accomplish Loops. Other crossing/dual path mechanics are level-specific.

        Each Zone in this game can be given two Chunk IDs to define as "Loops". These Chunks must be followed immediately by a visibly-identical copy, with the original having solidity filled in on the right half, and the copy having solidity filled in on the left half (For example, if Chunks 4 and 8 are "Loop" Chunks, they must be made solid on the right side, and must also have copies in Chunks 5 and 9, made solid on the left side). The original and copy are swapped back and forth in-game depending on which side the player is moving toward them from.

        When placing these Chunks into the Level Layout, they must be "flagged". This can be accomplished in SonED2 by hovering over the Chunk within the Level Layout and pressing spacebar. The Status view will then add the word "(Loop)" to the displayed Chunk ID. When "grabbing" a Chunk from the layout that has already been "flagged", drawing that Chunk to the layout will automatically draw it as "flagged".

        The current script set version does not support changing the Chunk IDs for "Loops"; any "Loops" drawn into a level must use the same Chunk IDs as the "Loops" in the original, unmodified level.



       Spin Tunnels

        This game uses a "Force Spin Object" to achieve forced-spin tunnels.

        These Objects will cause the player to enter a spinning state automatically, and will forceably apply velocity in a certain direction based on their settings.

        This is still standard spinning state, which allows jumping, and standing up when slowed to 0 velocity.



       Level Boundaries

        For the most part, Level Boundaries (the area within which the screen is allowed to scroll across the level during gameplay) are not dynamically altered in this game, meaning that the initial (and persistent) settings can be modified.

        However, the current script set version does not support this; Level Boundary data is not available for editing.



       Fixed and Dynamic Water Heights

        The current script set version does not support affecting "water"; Which levels do and don't contain water, the Y position of the water at level start, the events that change the Y position of the water, and the Y positions that the water is moved to are not available for editing.



       Tidal Tempest Water Currents

        The water currents in Tidal Tempest are controlled by standard Objects, which are movable and modifiable in SonED2's Object Editor.



       Quartz Quadrant Inside/Outside Background Swap

        Quartz Quadrant has two backgrounds that are swapped back and forth during gameplay to represent an "indoor" area and an "outdoor" area. This is accomplished by reloading layout, tile, and palette data based on the player's location.

        The current script set version does not support modifying this behavior; the locations, tileset, layout, and palette data are not available for editing.



       Stardust Speedway Interwinding "X" Paths

        The winding "X"-shaped paths in Stardust Speedway are similar to loops in that the Chunks must be "loop" flagged within the layout to function, and must be immediately followed by a visibly-identical copy with a secondary solidity setup.

        The primary Chunk must contain solidity leading uphill from the lower-left to the upper-right, with the copy containing solidity leading downhill from the upper-left to the lower-right.

        This is a level-specific mechanic.



       Stardust Speedway Path Swap Panels

        The dual-path sections of Stardust Speedway that are affected by the flipping blue sphere panels are similar to loops in that the Chunks must be "loop" flagged within the layout to function, and must be immediately followed by a visibly-identical copy with a secondary solidity setup.

        The primary Chunk must contain solidity for the "in front" portion, with the copy containing solidity for the "behind" portion.

        This is a level-specific mechanic.



       Metallic Madness In-Front/Behind Flipping Curtains

        The dual-path sections of Metallic Madness that allow travel in front of and behind the vertical flipping curtains are similar to loops in that the Chunks must be "loop" flagged within the layout to function, and must be immediately followed by a visibly-identical copy with a secondary solidity setup.

        The primary Chunk must contain solidity for the "in front" portion, with the copy containing solidity for the "behind" portion.

        This is a level-specific mechanic.



       Boss Arenas

        Boss events are triggered by standard Objects, which are movable in SonED2's Object Editor, but be aware that moving them or modifying the surrounding area could cause undesireable results in some cases.
