    Setup and Basic Usage
      This script set is "non-destructive". This means that the input game files will never need to be restored, as the Injection process never modifies them. Thus, all changes are applied to a clean copy of the game, and saved to a different file. The only time any of the initial steps will need to be repeated is if the separated data files have become corrupt, in which case, the Split will need to be performed again.


      One-Time Setup Steps
      • Unpack all files from this archive into their own folder. The full path name of this folder must not contain any spaces
      • Place in the same folder a copy of Revision 0 of "Sonic the Hedgehog" and give it the filename "sonic.bin"
      • Run the included Batch File "split.txt" using ROMulan or SonED2's built-in ROMulan functionality to retrieve the game's data


      Every Time Level Editing Steps
      • Using SonED2's "Open Project" option in the "File" menu, navigate to the "Projects" folder included with this fileset and open any of the included Project Files to edit its corresponding level
      • Make any changes using SonED2 as described in its own documentation
      • Use SonED2's "Save Project" option by selecting it in the "File" menu or pressing F3 to save the changes to their respective files (This is important as the edited data is SonED2-internal; ROMulan expects to find all necessary data in the files listed by the injector script)


      Other Modifications
      • With the proper knowledge of the game, other changes can be made by modifying the two Batch Files and/or the Injector File "inmain.txt" as described by the ROMulan documentation. For relevant information, see the "Game Notes" section.
      • Some game settings are already available for modification in the Injector File "inopts.txt". See the "Special Options" section for details.
      • The names of the Zones as presented on the "TitleCards" may be modified through the Injector File "incard.txt". See the "Level TitleCard Names" section for details.


      Every Time Finalization Steps
      • Run the included Batch File "inject.txt" using ROMulan or SonED2's built-in ROMulan functionality to inject the changes into the game
      • The file "inject.bin" will have been created and placed in the same folder as the "inject.txt" script. The process is now complete

    Game Notes
      Tile Paging and Shared Level Data

        One Zone of this game makes use of "Tile Paging", a system where a level's full Tileset is broken up into multiple "Pages" that can be swapped out during gameplay or shared between levels. This Zone is Green Hill Zone, which uses two Pages for its 8x8 Tiles, and shares Page 0 with the Title Screen.

        Tiles are shared between every Act of any one Zone. Modifying them for one Act will modify them for every other Act within the same Zone.

        Data-wise, "Scrap Brain Zone Act 3" is actually Labyrinth Zone Act 4. The same Tile data (but not the palette data) that is used for Labyrinth Zone is also loaded for Scrap Brain Zone Act 3.



       Animated Tiles and Palettes

        Certain Tiles and Palette entries are animated/cycled during gameplay. This behavior is not accounted for in the current script set version; these animations and the data that they use are not available for editing. Any modifications to these Tile or Palette entries will be overridden by these animations during gameplay.

        In this game, these Tiles use greater IDs than the last stored Tile within the Tile data, so they will be displayed in SonED2 as an "X" inside of a square (the "invalid Tile" marker).



       Loops and Crossing Paths

        This game uses a one-path collision system, meaning that Chunks within the level layout must be actively changed during gameplay to accomplish Loops. Other crossing path types are not supported by this game's specific implementation.

        Each Zone in this game can be given two Chunk IDs to define as "Loops". These Chunks must be followed immediately by a visibly-identical copy, with the original having solidity filled in on the right half, and the copy having solidity filled in on the left half (For example, if Chunks 4 and 8 are "Loop" Chunks, they must be made solid on the right side, and must also have copies in Chunks 5 and 9, made solid on the left side). The original and copy are swapped back and forth in-game depending on which side the player is moving toward them from.

        When placing these Chunks into the Level Layout, they must be "flagged". This can be accomplished in SonED2 by hovering over the Chunk within the Level Layout and pressing spacebar. The Status view will then add the word "(Loop)" to the displayed Chunk ID. When "grabbing" a Chunk from the layout that has already been "flagged", drawing that Chunk to the layout will automatically draw it as "flagged".

        If the Chunk IDs for "Loops" in a modified level are not the same as the IDs used in the unmodified level, the "        Loop and Spin Tunnel Chunk IDs" section of "inopts.txt" must be updated.



      Spin Tunnels

        Each Zone in this game can be given two Chunk IDs to define as "Spin Tunnels". These Chunks will force the player to spin, and apply rightward velocity if he decelerates to 0 while still in collision with the Chunk.

        If the Chunk IDs for "Spin Tunnels" in a modified level are not the same as the IDs used in the unmodified level, the "        Loop and Spin Tunnel Chunk IDs" section of "inopts.txt" must be updated.



      Level Boundaries

        Upper and lower Level Boundaries (the area within which the screen is allowed to scroll across the level during gameplay) are dynamically changed during gameplay in many cases, however, the leftmost and rightmost boundaries are not.

        Initial (and persistent) horizontal level start and end settings for each level can be changed by modifying the "        Level Size Array" section in "inopts.txt".

        Dynamically-altered vertical Level Boundaries are not accounted for in the current script set version; the level positions at which they are triggered, and the new settings that each of them applies are not available for editing.



      Labyrinth Zone Current Tunnels

        The Current Tunnels that carry the player along in Labyrinth Zone are triggered by hardcoded player position tests. This is not accounted for in the current script set version; the level areas that are regarded as Current Tunnels are not available for editing.



       Fixed and Dynamic Water Heights

        The current script set version does not support affecting "water"; Which levels do and don't contain water, the Y position of the water at level start, the events that change the Y position of the water, and the Y positions that the water is moved to are not available for editing.



       Boss Arenas

        Boss events are triggered by hardcoded player position tests, and the Boss Objects themselves test for and set absolute positions within the level. This is not accounted for in the current script set version; the level positions at which the Boss is triggered and the position values used by the Boss Object for movement are not available for editing.

        The level can, however, be made to end before the Boss area by decreasing the corresponding XEnd setting within the "        Level Size Array" section in "inopts.txt", and placing a Signpost Object at the new "end" location.


    Special Options
      Level Music Playlist

        The "Level Music Playlist" and "Secondary Music Playlist" sections of "inopts.txt" specify which songs play for which Zones.

        Both lists must be modified for the change to be complete. Within each list, find the entry that corresponds with the desired Zone, and replace the existing value with the ID number of the music that should be played for the Zone.

        The ID numbers entered in this section are the Song Numbers as they appear in the game's Sound Test.

        Note: The "Secondary Music Playlist" does not and should not contain a "Final Zone" entry.


       Level Boundaries

        The "Level Size Array" section of "inopts.txt" contains the listing that defines the initial Level Boundaries (the area within which the screen is allowed to scroll across the level during gameplay) for each level.

        Each Zone entry is laid out in the following manner:
            WORD	4, XStart, XEnd, YStart, YEnd, $60
        • 4
          		•  - This entry is unused, but required
        • XStart
          		•  - The leftmost world position that the screen is allowed to scroll to within the level. This value is only overridden by Boss Arenas.
        • XEnd
          		•  - The rightmost world position that the screen is allowed to scroll to within the level. This value is only overridden by Boss Arenas.
        • YStart
          		•  - The uppermost world position that the screen is allowed to scroll to within the level. This value is generally overridden throughout gameplay by dynamic screen relocation.
        • YEnd
          		•  - The lowermost world position that the screen is allowed to scroll to within the level. This value is generally overridden throughout gameplay by dynamic screen relocation.
        • $60
          		•  - This entry is unused, but required

        Level Y Looping can be achieved by entering "YLoop" into the "YStart" field, but for this to work, the level must be exactly $800 pixels (8 Chunks) in height, and the level must not be affected by dynamic Y boundary changes that will override this setting.


       Loop and Spin Tunnel Chunk IDs

        The "Loop and Spin Tunnel Chunk IDs" section of "inopts.txt" defines the Chunks that act as Loops and Spin Tunnels for each Zone.

        Each Zone entry is laid out in the following manner:
            	BYTE Loop1+LoopFlag, Loop2+LoopFlag, Tunnel1, Tunnel2
        • Loop1
          		•  - The ID number of a Chunk to define as a "Loop". This can only be a standard Loop, not a "P"-shaped Loop.
        • Loop2
          		•  - The ID number of a Chunk to define as a "Loop". This can be a standard loop, or a "P"-shaped Loop (as seen in Star Light Zone).
        • Tunnel1
          		•  - This is the ID number of a Chunk to define as a "Spin Tunnel".
        • Tunnel2
          		•  - This is the ID number of a Chunk to define as a "Spin Tunnel".

        The value "LoopFlag" must be added to both "Loop" entries.

        When no Loop Chunk exists, or when only one Loop Chunk is desired, unused entries should be filled with "NoLoop".

        When no Spin Tunnel Chunk exists, or when only one Spin Tunnel Chunk is desired, unused entries should be filled with "NoTunnel".


       Starting Level and Level Progression Order

        The "Starting Level" section of "inopts.txt" determines the level on which the game starts, after pressing Start on the TitleScreen.

        To change the Starting Level, find the line containing the comment "Zone and Act to go to when starting a new game", and change the two values to reflect the Zone and Act IDs of the desired level.

        The ""Next Level" array" section of "inopts.txt" determines the order in which the levels are presented during gameplay.

        This list is not laid out in the order in which the levels will be played, but rather, contains an entry for each level in the game in order of Zone and Act ID. The level defined by the entry will be played after the level that corresponds with the entry has been completed (Ex: The first entry in the list corresponds with Green Hill Zone Act 1. The data in the entry specifies that Green Hill Zone Act 2 will be played after Green Hill Zone Act 1 is complete).

        The Zone and Act name "Defines" that are listed in the "Data Defines" section can be used for either of these settings.


      Cheat Codes

        The "Cheat Codes" section of "inopts.txt" defines the buttonpresses that are required to activate the game's Cheat Codes.

        The Cheat Codes are defined by entering the names of any of the "Input State Bits" "Defines" in any order, any number of times.

        The last buttonpress in the code must be followed by ", Code_End, Code_End2" to mark the end.

        To effectively disable the code, insert "Input_Invalid" as a "buttonpress" within the code.


     Level TitleCard Names
      The "TCardMapData" segment in "incard.txt" contains the standard Sonic 1 Sprite Mappings data that make up the Zone names and other TitleCard elements.

      As all available letters (there is no J, Q, V, W, or X) are loaded into VRAM every time, changing the names presented by the TitleCard is as simple as changing each mapping to call on the desired letter art.

      Each mapping starts with a "BYTE" that defines the number of entries that it contains. The entries themselves are formatted as follows:

          BYTE	YOff, MWidth?? + MHeight??, 0, TCTilePos_?, XOff

      • YOff
        		•  - The Y offset from the TitleCard's position at which this letter should be placed. Generally, this should remain "$F8".
      • MWidth?? + MHeight??
        		•  - Add together the names of the Width and Height "Defines" that describe the pixel size of the desired letter.
      • 0
        		•  - This value should remain "0".
      • TCTilePos_?
        		•  - The name of the "Define" that corresponds with the desired letter.
      • XOff
        		•  - The X offset from the TitleCard's position at which this letter should be placed.

      The "Defines" used for these entries are listed at the top of the file.

      To effectively insert a space "character", simply use the "XOff" entry to place the character after the "space" farther away from the one before.
