SonED2 - Standard Level Project Format


This type of project allows for editing Tiles/Blocks/Chunks, Level Layout, and Object Layout for standard gameplay levels. The format varies slightly at several points depending on the type of data being referenced, so it is important to pay attention to how the fields are used under different circumstances. For complete and functional example usage, please see the Project Files within the included ROMulan packages.

Regardless of their exact contents, the following sections are always used in every Standard Level Project File, in this order:

    [Project Type, Level ID, and Object Def References]
      The structure of this section never changes. It contains the following general setup data:

      Type:
        The first line in any Project File is the "Type" identifier. This Project's "Type" should read "Level"

          Type: Level

      Zone ID:
      Act ID:
        These are used internally by SonED2 in cases such as assigning more than one set of art and/or state settings to a single Object ID in the Object Listing, which are determined by the level being edited. These values should be set equal to the Zone ID and Act ID of the level within its parent game. Remember that these refer to the game-internal values, not the gameplay order of the levels, or the values shown on-screen.

          Zone ID: 0
Act ID:  0

      Object Def:
        This is the path and filename of the Object Listing file that contains graphic and settings information for displaying the objects in this level. This filename must not be accompanied by an extension; the extension ".lst" will be added by SonED2 when loading the list, and this filename will also be used to generate the filename strings for each of the accompanying graphics files

          Object Def: ObjDef\s1obj

      Object Palette:
        This is the path and filename of a PCX or TGA graphics file from which to load extra colors for displaying "Object" graphics in SonED2. The colors from indexes 192-255 will be loaded from this image and made available for use by Object graphics loaded by an Object Listing File. Normally it is preferable to store those graphics such that they use indexes 0-63, and thus, are subjected to the level palette the same way they are in-game, however, these extra colors are available in case any special-case global visual aid graphics are desired, or in the case of Objects that don't actually use the Genesis/Megadrive palette, such as Chaotix's 32X Sprites

          Object Palette: ObjDef\s1obj.tga


     [8x8 Tiles]
      This section specifies how to load the 8x8 Tile graphic data:

      8x8 Compression:
        This entry specifies the type of compression that is used on the 8x8 Tile graphic data

          8x8 Compression: 3

      Num 8x8 Pages:
        This value specifies the number of 8x8 Tile graphic "pages" that should be loaded. Each "page" is stored as a separate file, so this specifies the number of filenames that will be supplied:

          Num 8x8 Pages: 2

        8x8 Tiles Page ?:
          This is a path and filename of the 8x8 Tile graphic data file to load

            8x8 Tiles Page 0: ..\Levels\GHZ\Tile\8_0.bin


    [Blocks]
      This section specifies how to load the 16x16 Block Mappings data:

      16x16 Compression:
        This entry specifies the type of compression that is used on the 16x16 Block Mappings data

          16x16 Compression: 2

      Num 16x16 Pages:
        This value specifies the number of 16x16 Block Mappings "pages" that should be loaded. Each "page" is stored as a separate file, so this specifies the number of filenames that will be supplied:

          Num 16x16 Pages: 1

        Page? 8x8 Page:
        16x16 Tiles Page ?:
          Each 16x16 Block Mappings "page" will need both of these entries, which specify ID number the 8x8 Tile Page that the Blocks reference, and the filename of the Block data file, respectively

            Page0 8x8 Page:     0
16x16 Tiles Page 0: ..\Levels\GHZ\Tile\16.bin


    [Metatiles/Chunks]
      This section specifies how to load the MetaTile Mappings data:

      Metatile Size:
        This entry specifies the height and width size (in pixels) of the Metatiles/Chunks. Valid sizes are either "128" or "256"

          Metatile Size: 256

      Metatile Compression:
        This entry specifies the type of compression that is used on the Metatile data

          Metatile Compression: 2

      Num Metatile Pages:
        This value specifies the number of Metatile "pages" that should be loaded. Each "page" is stored as a separate file, so this specifies the number of filenames that will be supplied:

          Num Metatile Pages: 1

        Page? 16x16 Page:
        Metatile Tiles Page ?:
          Each Metatile "page" will need both of these entries, which specify ID number the 16x16 Block Mapping Page that the Metatiles reference, and the filename of the Metatile data file, respectively

            Page0 16x16 Page:      0
Metatile Tiles Page 0: ..\Levels\GHZ\Tile\256.bin


    [Level Layout]
      This section specifies how to load the Level Layout data. The first two entries are always the same, while the following entries differ depending on the "Layout Format" setting:

      Layout Format:
        This entry specifies the format in which the (uncompressed) level data is stored. Valid settings are:

          0 - Sonic 1 / Sonic CD
          1 - Sonic 2
          2 - Sonic 3 (& Knuckles)
          3 - Knuckles' Chaotix / Sonic Crackers

          Layout Format:         0

      Layout Compression:
        This entry specifies the type of compression that is used on the Level Layout data

          Layout Compression:    0

      (Sonic 1 / Sonic CD)
        These games use the same data format, and store foreground and background data separately:

        FG Level Layout:
        BG 128x128 Page:
        BG Level Layout:
          These entries specify the path and filename of the file containing the data for the foreground (Plane A), the Metatile page that the background draws its tiles from, and the path and filename of the file containing the data for the background (Plane B), respectively
            FG Level Layout:       ..\Levels\GHZ\FG\1.bin
BG 128x128 Page:       0
BG Level Layout:       ..\Levels\GHZ\BG\1.bin


      (Sonic 2)
      (Sonic 3 (& Knuckles))
        These games use different data formats for Level Layout, but they both store foreground and background data together:

        BG 128x128 Page:
        Level Layout:
          These entries specify the Metatile page that the background draws its tiles from, and the path and filename of the one file containing the data for the both the foreground and background (Planes A and B), respectively
            BG 128x128 Page:       0
Level Layout:       	..\Levels\00_EHZ\Layout\1.bin


      (Knuckles' Chaotix / Sonic Crackers)
        These games use the same data format. While they store foreground and background data separately just as with Sonic 1, this format is different in that the Layout plane sizes are stored separately from the Layout data itself:

        FG Size:
        FG Level Layout:
        BG 128x128 Page:
        BG Size:
        BG Level Layout:
          These entries specify the following information, in this order- the path and filename of the file containing the height and width size data for the foreground (Plane A), the path and filename of the file containing the layout data for the foreground (Plane A), the Metatile page that the background draws its tiles from, the path and filename of the file containing the height and width size data for the background (Plane B), and the path and filename of the file containing the layout data for the background (Plane B), respectively
            FG Size:              ..\..\Levels\BBZ\FG\Size01MN.bin
FG Level Layout:      ..\..\Levels\BBZ\FG\1MN.bin
BG 128x128 Page:      2
BG Size:              ..\..\Levels\BBZ\BG\Size01MN.bin
BG Level Layout:      ..\..\Levels\BBZ\BG\12MN.bin


    [Object Layout]
      This section specifies how to load the Object Layout data. The first entry is always the same, while the following entries differ depending on the "Objects Format" setting:

      Objects Format:
        This entry specifies the format in which the Object Layout data is stored. Valid settings are:

          0 - No Objects
          1 - Sonic 1
          2 - Sonic CD
          3 - Sonic 2
          4 - Sonic 3 (& Knuckles)
          5 - Knuckles' Chaotix / Sonic Crackers

          Objects Format:         0

      (No Objects)
        This setting should be used when there are no Objects to load. In this case, there are no other entries in this section.

      (Sonic 1)
      (Sonic 2)
      (Sonic 3 (& Knuckles))
      (Knuckles' Chaotix / Sonic Crackers)
        These settings use different data formats, but each of them only requires the name of the data file to be entered:

        Objects:
          This entry specifies the path and filename of the file containing the Object Layout data
            Objects:       ..\Levels\GHZ\Obj\1.bin

      (Sonic CD)
        Unlike the others, this setting requires two entries:

        Objects Time:
          This entry specifies the time period of the Objects that should be displayed by default when the Project File is loaded. Valid settings are:

            A - "Present"
            B - "Past"
            C - "Future"
            D - "Future"

          Note that "C" and "D" have the same effect, as Objects in this format only use one flag for "Future", rather than separating "Good Future" and "Bad Future".

            Objects Time:         A


        Objects:
          This entry specifies the path and filename of the file containing the Object Layout data
            Objects:       ..\Levels\GHZ\Obj\1.bin


     [Rings Layout]
      This section specifies how to load the Ring Layout data. The first entry is always the same, while the following entries differ depending on the "Rings Format" setting:

      Rings Format:
        This entry specifies the format in which the Ring Layout data is stored. Valid settings are:

          0 - No Separate Rings Layout
          1 - Sonic 2
          2 - Sonic 3 (& Knuckles)

      (No Separate Rings Layout)
        This setting should be used when the target game does not store Rings separately from other Objects, or when there are no Rings to load at all. In this case, there are no other entries in this section.

      (Sonic 2)
      (Sonic 3 (& Knuckles))
        These settings use different data formats, but each of them only requires the name of the data file to be entered:

        Rings:
          This entry specifies the path and filename of the file containing the Ring Layout data
            Rings:                 ..\Levels\0C_CNZ\Rng\1.bin


     ["Misc" Layout]
      This section specifies how to load the "Misc" Objects Layout data. The first entry is always the same, while the following entries differ depending on the "Misc Format" setting:

      Misc Format:
        This entry specifies the format in which the "Misc" Object Layout data is stored. Valid settings are:

          0 - No "Misc" Objects
          1 - Sonic 2 Casino Night Zone Bumpers

      (No "Misc" Objects)
        This setting should be used when the target game does not store certain special Objects separately from other Objects, or when there are no special Objects to load at all. In this case, there are no other entries in this section.

      (Sonic 2 Casino Night Zone Bumpers)
        This setting only requires the name of the data file to be entered:

        Misc:
          This entry specifies the path and filename of the file containing the "Misc" Object Layout data
            Misc:                  ..\Levels\0C_CNZ\Obj\Bump1.bin


     [Player Start Positions]
      This section specifies how to load the Player Start Positions data. The graphics that represent these positions within the editor are loaded by the Object Listing File

      Num Start Positions:
        This entry specifies the number of Player Start Positions that should be loaded. The value entered here determines the number of entries that should follow:

          Num Start Positions:   1

        Start Position ?:
          This entry specifies the path and filename of the file containing the Player Start Position data

            Start Position 0:      ..\Levels\GHZ\Start\1.bin


    [Palettes]
      This section specifies how to load the Palette Color Data. The Genesis/Megadrive palette only has 64 entries, but an additional color space of 64 is available in the editor for creating "underwater" and other accessory palettes that can be flip/flopped with the primary palette during editing

      Palette data is typically stored in pieces, such as the usual separation between the "Main" palette section (player and global objects, indexes 0-15, usually unchanging) and the "Level" palette section (indexes 16-63, different for every level), so, generally more than one file will need to be loaded at a time

      Number of files:
        This entry specifies the number of Palette Color Data files to load

          Number of files: 2

        The data for loading each palette section is listed immediately below in the following format:

        Start index:
          This entry specifies where to place the incoming data in the palette. The value entered here is the palette index into which the data for the first color will be placed

            Start index:        0

        Number of entries:
          This entry specifies the number of palette color entries that should be loaded from the file

            Number of entries: 16

        File:
          This entry specifies the path and filename of the file containing the Palette Color Data

            File:              ..\Palettes\Sonic.bin


     [Collision Maps]
      This section specifies how to load the Collision Height Map and Angle data

      Collision Map Format:
        This entry specifies the format in which the Collision Height Map data is stored. Valid settings are:

          0 - Sonic 1, CD, 2, 3 (& Knuckles)
          1 - Knuckles' Chaotix / Sonic Crackers

          Collision Map Format:    0

      Collision Array:
        This entry specifies the path and filename of the file containing the normal-oriented Collision Height Map data

          Collision Array:         ..\Collisn\HMaps.bin

      Rotated Collision Array:
        This entry specifies the path and filename of the file containing the 90-degree rotated version of the Collision Height Map data

          Rotated Collision Array: ..\Collisn\HMapsR.bin

      Angle Array:
        This entry specifies the path and filename of the file containing the Angles data

          Angle Array:             ..\Collisn\Angles.bin


     [Collision Indexes]
      This section specifies how to load the Collision Index data

      Collision Index Format:
        This entry specifies the format in which the Collision Index data is stored. Valid settings are:

          0 - Sonic 1 / Sonic CD
          1 - Sonic 2
          2 - Sonic 3
          3 - Sonic & Knuckles
          4 - Knuckles' Chaotix / Sonic Crackers

          Collision Index Format:      0

      Collision Index Compression:
        This entry specifies the type of compression that is used on the Collision Index data

          Collision Index Compression: 0


      (Sonic 1 / Sonic CD)
      (Sonic 3)
      (Sonic & Knuckles)
      (Knuckles' Chaotix / Sonic Crackers)
        These games use different data formats, but each of them only requires the name of the data file to be entered:

        Collision Index:
          This entry specifies the path and filename of the file that contains the Collision Index data for both Path 0 and (when applicable) Path 1

            Collision Index:             ..\Levels\GHZ\HMIndex.bin


      (Sonic 2)
        This game stores separate data for each of the two collision paths, and so requires two data file name entries:

        Collision Index 0:
        Collision Index 1:
          These entries specify the paths and filenames of the files that contain the Collision Index data for Path 0 and Path 1, respectively

            Collision Index 0: ..\Levels\0C_CNZ\Collisn\0.bin
Collision Index 1: ..\Levels\0C_CNZ\Collisn\1.bin


    [Game Name]
      This section allows a Game Name to be given to the Project, which will be displayed in the SonED2 titlebar along with the path and filename of the Project File

      Game Name:
        This entry is a text string that names the game that this Project File was created for

          Game Name: Sonic 1
