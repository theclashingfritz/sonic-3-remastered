Knuckles' Chaotix JU/(5) ROMulan Package


The files contained herein are for use with ROMulan and SonED2, to provide a means of editing the levels of the Japanese/USA-region (region code "5") release of Knuckles' Chaotix.

Please visit the official forum for discussion on mods made using this package, to submit additions to the scripts, or to report any errors in the official distribution.

    Setup and Basic Usage
      This script set is "non-destructive". This means that the input game file will never need to be restored, as the Injection process never modifies it. Thus, all changes are applied to a clean copy of the game, and saved to a different file. The only time any of the initial steps will need to be repeated is if the separated data files have become corrupt, in which case, the Split will need to be performed again.


      One-Time Setup Steps
      • Unpack all files from this archive into their own folder. The full path name of this folder must not contain any spaces
      • Place in the same folder a copy of the Japanese/USA-region (code "5") release of Knuckles' Chaotix and give it the filename "kc.bin"
      • Run the included Batch File "split.txt" using ROMulan or SonED2's built-in ROMulan functionality to retrieve the game's data


      Every Time Level Editing Steps
      • Using SonED2's "Open Project" option in the "File" menu, navigate to the "Projects" folder included with this fileset and open any of the included Project Files to edit its corresponding level
      • Make any changes using SonED2 as described in its own documentation
      • Use SonED2's "Save Project" option by selecting it in the "File" menu or pressing F3 to save the changes to their respective files (This is important as the edited data is SonED2-internal; ROMulan expects to find all necessary data in the files listed by the injector script)

        Be aware that modifying some levels' Tilesets will effectively modify the tilesets of certain other levels, but not all levels within the same zone. See the "        Game Notes" section for more information.


      Other Modifications
      • With the proper knowledge of the game, other changes can be made by modifying the two Batch Files and/or the Injector File "injmain.txt" as described by the ROMulan documentation. For relevant information, see the "Game Notes" section.
      • Some game settings are already available for modification in the Injector File "inopts.txt". See the "Special Options" section for details.


      Every Time Finalization Steps
      • Run the included Batch File "inject.txt" using ROMulan or SonED2's built-in ROMulan functionality to inject the changes into the game
      • The file "inject.bin" will have been created and placed in the same folder as the "inject.txt" script. The process is now complete

    Game Notes
      Tile Paging and Shared Level Data

        This game makes use of "Tile Paging", a system where a level's full Tileset is broken up into multiple "Pages" that can be swapped out during gameplay or shared between levels.

        The shared Tile Pages (and occasionally, certain other data) used in this game are used erratically- Pages are shared between acts and times-of-day in a way that is specifically tailored to the way that the commercial release of the game game was built; there is no generic pattern that can easily be worked with (such as "all acts within a zone always use the same Page 0 for Daytime Tiles").

        When modifying tilesets for this game, it is important to know which levels will be affected by the modifications. The following information provides details on using either of the two listed filetypes to determine how the data is currently paged and shared, and how to modify the structure so that the paging and sharing will better suit any specific modifications:

        • Project Files

            Check the Project File for the level being edited. The naming system for the data files includes descriptions of their use:

          • Acts number 0, 1, 2, 3, 4, 5, and 6
          • Times of day are in order of "M", "D", "S", and "N"

            As an example, the name "15MN.bin" means that this data is used in acts spanning 1 through 5, and times of day spanning Morning through Night.

        • Injector File

            Check the pointers listings in the Injector File "injmain.txt" by searching for the following data:

          • "LevelTile_Ptrs" - References pointers to the 8x8 Tile Page loading data

              This data is laid out in the following format:

                  DWORD TilesPage0
DWORD TilesPage1
DWORD 0
DWORD TilesPage2
DWORD TilesPage3
DWORD $FFFFFFFF

              • TilesPage?
                		•  - The name of the label immediately preceding the 8x8 Tile data to load for the given page
              • 0
                		•  - Terminates loading for the "Foreground" Pages and begins loading for the "Background" Pages
              • $FFFFFFFF
                		•  - Terminates the tileset

              Any number of Pages can be loaded for each of the Foreground or Background. If the "0" is omitted, the Background does not use its own Pages (it shares Tile Pages with the Foreground). The "$FFFFFFFF" cannot be omitted.

          • "Level_Index"- References pointers to the 16x16 Block Page, 128x128 Chunk Page, and Level Layout loading data

              This data is laid out in the following format:

                  DWORD BlocksPage0
DWORD BlocksPage1+LastPage
DWORD BlocksPage2
DWORD BlocksPage3+LastPage
DWORD ChunksPage0
DWORD ChunksPage1+LastPage
DWORD ChunksPage2
DWORD ChunksPage3+LastPage
DWORD FGLayout
DWORD BGLayout

              • BlocksPage?
                		•  - The name of the label immediately preceding the 16x16 Block data to load for the given page
              • ChunksPage?
                		•  - The name of the label immediately preceding the 128x128 Chunk data to load for the given page
              • ??Layout
                		•  - The names of the labels immediately preceding the Foreground layout data and Background layout data, respectively

              Any number of Block and Chunk Pages can be loaded for each of the Foreground and Background. If the Background should not load its own Block or Chunk pages, and should instead share Pages with the Foreground, "+NoNewLayer" should be added to the line (as in "BlocksPage1+LastPage+NoNewLayer").

          • "LevSize_Data"- References pointers to the level initial camera (and Player Start) position, layout size, and scroll boundaries

              This data is laid out in the following format:

                  include	"PlaneStart.bin"
include	"PlaneSize.bin"
WORD	X2
WORD	X1
WORD	Y2
WORD	Y1

              • "PlaneStart.bin"
                		•  - The path and filename of the data file containing the initial screen start location for the level. Also serves to generate Player Start position
              • "PlaneSize.bin"
                		•  - The path and filename of the data file containing the map layout dimensions for the level
              • X2
                		•  - The maximum X position that the screen can scroll to within this level during gameplay
              • X1
                		•  - The minimum X position that the screen can scroll to within this level during gameplay
              • Y2
                		•  - The maximum Y position that the screen can scroll to within this level during gameplay
              • Y1
                		•  - The minimum Y position that the screen can scroll to within this level during gameplay

          • "MD_Palette_Index"- References pointers to the level palette data

              This data is laid out in the following format:

                  include	"Pal.bin"

              • "Pal.bin"
                		•  - The path and filename of the data file containing the level's palette colors

          • "Solid_Index"- References pointers to the collision index data

              This data is laid out in the following format:

                  include	"Solid.bin"

              • "Solid.bin"
                		•  - The path and filename of the data file containing the level's collision index data



       Animated Tiles and Palettes

        Certain Tiles and Palette entries are animated/cycled during gameplay. This behavior is not accounted for in the current script set version; these animations and the data that they use are not available for editing. Any modifications to these Tile or Palette entries will be overridden by these animations during gameplay.

        In this game, the Tile data contains number and/or special character graphics at these positions, which is what will be displayed in SonED2.



       Loops, Crossing Paths, and Plane Priority

        This game uses a two-path collision system, and so, it supports level paths that cross freely by means of a "Path Swapper Object".

        However, unlike other 2D Sonic the Hedgehog games, the player sprites are handled by the 32x, and 32x graphics may only be drawn either above all Megadrive/Genesis graphics, or below all Megadrive/Genesis graphics. This means that the player cannot be placed between Plane A and Plane B, or between High and Low Tile Priorities. In effect, the Player must be displayed above all level graphics to be seen at all, and can never go "behind" anything that is not also a 32x sprite.

        In short, while standard loops and other path crosses are technically possible, they will not look right in cases where part of the level should be overlapping the player.



       Level Boundaries

        For the most part, Level Boundaries (the area within which the screen is allowed to scroll across the level during gameplay) are not dynamically altered in this game, meaning that the initial (and persistent) settings can be modified.

        See the "LevSize_Data" description in the "Tile Paging and Shared Level Data" section above for editing information.



       Dynamic Level Changes

        Certain levels are hardcoded to change certain aspects of the level during gameplay, most notably palettes. This behavior is not accounted for in the current script set version; the data loaded mid-level is not available for editing, and there is no support for modifying or cancelling the events.



       Boss Arenas

        Boss events are triggered by standard Objects, which are movable in SonED2's Object Editor, but be aware that moving them or modifying the surrounding area could cause undesireable results in some cases.



       Emulator Compatibility with Modified Game

        This game uses a special timing mode of the 32x that emulators must support in order to run the game correctly. If the emulator is not using the correct mode, there may be severe graphical errors and crashes.

        Some emulators, such as Kega, only enter this mode when they realize that they are running this game, which is handled by performing a checksum on the game and comparing it to the known checksum of an unmodified version. Modifying the game changes the checksum, and thus, renders it unrecognizeable.

        To correctly play the modified game in Kega, use the "Set Config..." option in the "Options" menu. Once the config window appears, select the "32x" tab, and search for a checkbox that reads "Fast 32x Timing". This box must be checked in order for the game to behave correctly. This option only exists in newer Kega releases.


    Special Options
      Level Music Playlist

        The "Level Music Playlist" section of "inopts.txt" specifies which songs play for which Zones.

        Within the list, find the entry that corresponds with the desired Zone, and replace the existing value with the ID number of the music that should be played for the Zone.

        The ID numbers entered in this section must be calculated by counting their positions in the Sound Test Menu, with "--- SOUND TEST ---" being 0, "DOOR INTO SUMMER" being 1, etc.


       Cheat Codes

        The "Cheat Codes" section of "inopts.txt" defines the "Color Test" settings that must be entered to activate the game's Cheat Codes.

        The "Code_LevelSelectDebug" section defines the settings that must be entered to activate both the Level Select menu and Debug Mode. This code is input using the first and second color entries on the left side.

        The "Code_AmyInSoundTest" section defines the settings that must be entered to cause Amy to appear on the Sound Test screen. This code is input using the second and third color entries on the left side.

        The values that must be entered in this section represent the four-digit value that appears at the top-left of the screen, not the three color components that are being directly modified. The first and second entries for each code in this section correspond directly with the first and second color entries used for the code, as described above.
