The files contained herein are for use with ROMulan and SonED2, to provide a means of editing the levels of Revision 1 of "Sonic the Hedgehog 2".

    Setup and Basic Usage
      This script set is "non-destructive". This means that the input game files will never need to be restored, as the Injection process never modifies them. Thus, all changes are applied to a clean copy of the game, and saved to a different file. The only time any of the initial steps will need to be repeated is if the separated data files have become corrupt, in which case, the Split will need to be performed again.


      One-Time Setup Steps
      • Unpack all files from this archive into their own folder. The full path name of this folder must not contain any spaces
      • Place in the same folder a copy of Revision 1 of "Sonic the Hedgehog 2" and give it the filename "sonic2.bin"
      • Run the included Batch File "split.txt" using ROMulan or SonED2's built-in ROMulan functionality to retrieve the game's data


      Every Time Level Editing Steps
      • Using SonED2's "Open Project" option in the "File" menu, navigate to the "Projects" folder included with this fileset and open any of the included Project Files to edit its corresponding level
      • Make any changes using SonED2 as described in its own documentation
      • Use SonED2's "Save Project" option by selecting it in the "File" menu or pressing F3 to save the changes to their respective files (This is important as the edited data is SonED2-internal; ROMulan expects to find all necessary data in the files listed by the injector script)


      Other Modifications
      • With the proper knowledge of the game, other changes can be made by modifying the two Batch Files and/or the Injector File "inmain.txt" as described by the ROMulan documentation. For relevant information, see the "Game Notes" section.
      • Some game settings are already available for modification in the Injector File "inopts.txt". See the "Special Options" section for details.
      • The names of the Zones as presented on the "TitleCards" may be modified through the Injector File "incard.txt". See the "Level TitleCard Names" section for details.


      Every Time Finalization Steps
      • Run the included Batch File "inject.txt" using ROMulan or SonED2's built-in ROMulan functionality to inject the changes into the game
      • The file "inject.bin" will have been created and placed in the same folder as the "inject.txt" script. The process is now complete

    Game Notes
      Shared Level Data and Unused Level Slots

        Tiles are shared between every Act of any one Zone. Modifying them for one Act will modify them for every other Act within the same Zone.

        In an unmodified game, Emerald Hill Zone and Wing Fortress Zone share Tile data with Hill Top Zone and Sky Chase Zone, respectively, and are overlayed with special patch data to replace only certain Tiles. This would normally cause modifications to either Zone to appear in the other, however, this script set duplicates these Tilesets and causes each Zone to use its own. It also pre-applies the Tileset patches and disables in-game patching to allow for proper viewing and editing of the patched levels.

        An unmodified game also has space allocated for 5 unused Zones, and 4 unused second acts. Normally, these levels load most of their data from existing, used levels, however, this script set duplicates the re-used data and causes these unused Zones and Acts to each use their own dataset, and also includes Project Files for modifying these levels. They may now be used to extend the game, and each of the unused Zones may be given their own set of level graphics.

        The "        Next Level" array" section of "inopts.txt" may be used to add these new levels to the gameplay progression.



      Animated Tiles and Palettes

        Certain Tiles and Palette entries are animated/cycled during gameplay. This behavior is not accounted for in the current script set version; these animations and the data that they use are not available for editing. Any modifications to these Tile or Palette entries will be overridden by these animations during gameplay.

        In this game, the Tile data contains blank "graphics" at these positions, which is what will be displayed in SonED2.



       Loops, Crossing Paths, and Plane Priority

        This game uses a two-path collision system, and so, it supports level paths that cross freely by means of a "Path Swapper Object".

        These Objects can be used to cause the player to switch back and forth between collision with "Path 0" and "Path 1", which may be viewed and edited independently in SonED2.

        The same Objects may also change the player's "priority", causing him to be displayed either in front of or behind the "High" level plane for effects such as loops or other path crosses that overlap the player, or even walls that can be crossed both from in front and behind.



       Spin Tunnels

        This game uses a "Spin Tunnel Object" to achieve forced-spin tunnels.

        These Objects will cause the player to enter a spinning state that prevents jumping or standing up, and which will apply horizontal velocity in the direction that he is facing if he decelerates to 0, until he crosses another Object of the same type that is set to disable this mode.



       Level Boundaries

        For the most part, Level Boundaries (the area within which the screen is allowed to scroll across the level during gameplay) are not dynamically altered in this game, meaning that the initial (and persistent) settings can be modified.

        The "        Level Size Array" section of "inopts.txt" can be used to modify these settings.



      Hill Top Zone Earthquakes

        The Earthquake events in Hill Top Zone are triggered by hardcoded player position tests. This is not accounted for in the current script set version; the level areas that trigger the Earthquakes are not available for editing.

        The floor and lava level sections that rise and fall during these events are stored on Plane B (the "background"), which is how they are able to move separately from Plane A (the "foreground"), and they can be modified there.



       Fixed and Dynamic Water Heights

        The current script set version does not support affecting "water"; Which levels do and don't contain water, the Y position of the water at level start, the events that change the Y position of the water, and the Y positions that the water is moved to are not available for editing.



       Boss Arenas

        Boss events are triggered by hardcoded player position tests, and the Boss Objects themselves test for and set absolute positions within the level. This is not accounted for in the current script set version; the level positions at which the Boss is triggered and the position values used by the Boss Object for movement are not available for editing.

        The level can, however, be made to end before the Boss area by decreasing the corresponding XEnd setting within the "        Level Size Array" section in "inopts.txt", and placing a Signpost Object at the new "end" location.


    Special Options
      Level Music Playlist

        The "Level Music Playlist" section of "inopts.txt" specifies which songs play for which Zones.

        Within the list, find the entry that corresponds with the desired Zone, and replace the existing value with the ID number of the music that should be played for the Zone.

        The ID numbers entered in this section are the Song Numbers as they appear in the game's Sound Test, plus $80. Entering the song's Sound Test ID and adding " + MakeMusicID" will generate the appropriate value.


       Level Boundaries

        The "Level Size Array" section of "inopts.txt" contains the listing that defines the initial Level Boundaries (the area within which the screen is allowed to scroll across the level during gameplay) for each level.

        Each Zone entry is laid out in the following manner:
            WORD	XStart, XEnd, YStart, YEnd
        • XStart
          		•  - The leftmost world position that the screen is allowed to scroll to within the level
        • XEnd
          		•  - The rightmost world position that the screen is allowed to scroll to within the level
        • YStart
          		•  - The uppermost world position that the screen is allowed to scroll to within the level
        • YEnd
          		•  - The lowermost world position that the screen is allowed to scroll to within the level

        Level Y Looping can be achieved by entering "YLoop" into the "YStart" field, but for this to work, the level must be exactly $800 pixels (16 Chunks) in height.


       Starting Level and Level Progression Order

        The "Starting Level" section of "inopts.txt" determines the level on which the game starts, after pressing Start on the TitleScreen.

        To change the Starting Level, find the line containing the comment "Zone and Act to go to when starting a new game", and change the two values to reflect the Zone and Act IDs of the desired level.

        The ""Next Level" array" section of "inopts.txt" determines the order in which the levels are presented during gameplay.

        This list is not laid out in the order in which the levels will be played, but rather, contains an entry for each level in the game in order of Zone and Act ID. The level defined by the entry will be played after the level that corresponds with the entry has been completed (Ex: The first entry in the list corresponds with Emerald Hill Zone Act 1. The data in the entry specifies that Emerald Hill Zone Act 2 will be played after Emerald Hill Zone Act 1 is complete).

        The Zone and Act name "Defines" that are listed in the "Data Defines" section can be used for either of these settings.


      Cheat Codes

        The "Cheat Codes" section of "inopts.txt" defines the Sound Test IDs that must be played to activate the game's Cheat Codes.

        The Cheat Codes are defined by entering the exact Sound Test IDs numbers that must be played, in the order that they must be used.

        The last Sound Test ID in the code must be followed by ", Code_End" to mark the end.

        To effectively disable any code, insert "Code_Invalid" in place of a Sound Test ID within the code.


     Level TitleCard Names
      Letter Art Loading

        This game loads only the letters that are needed to display the current TitleCard, so it is necessary to specify which letters those are.

        The "Load Data" section of "incard.txt" contains the data for loading the letter graphics for each TitleCard. To specify the letters that should be loaded for any level, simply find its corresponding data row, and use it to list the names of the Defines that correspond with the desired letters, followed by ", TCLoad_End" to mark the end of the list.

        Letters "E", "N", "O", and "Z" are always loaded by default, and should not be included in the letter loading lists.

        The Defines in the "Letter Graphics VRAM Tile Offset Data" area assist in keeping track of where the letters are loaded to in VRAM, as this is affected by changes to the letter loading list. For the same reason, there is a list of these Defines for each Zone.

        The first letter that is loaded will always use the value "TCCustomStart", and the position of every letter following can be calculated by adding the Define name for the previous letter's position to the Define name for the previous letter's size.


       Name Display

        The "TCardMapData" segment in "incard.txt" contains the standard Sonic 2 Sprite Mappings data that build the displayed Zone names and other TitleCard elements.

        Once the letter loading data and VRAM offset Defines are set up, changing the names presented by the TitleCard is as simple as changing each mapping to call on the desired letter art.

        Each mapping starts with a "WORD" that defines the number of entries that it contains. The entries themselves are formatted as follows:

            BYTE	MWidth?? + MHeight??, TCTilePos_? + MPHigh, TCTilePos_? / 2 + MPHigh, XOff

        • MWidth?? + MHeight??
          		•  - Add together the names of the Width and Height "Defines" that describe the pixel size of the desired letter. A Y offset can be applied to the letter by also adding "YOff * 16 + " to the beginning of the entry, where "YOff" is the value by which to offset the letter from the Titlecard position on the Y axis. Generally, there should be no Y offset, however.
        • TCTilePos_? + MPHigh
          		•  - The name of the "Define" that corresponds with the desired letter, plus the define "MPHigh" to give the letter "High" priority.
        • TCTilePos_? / 2 + MPHigh
          		•  - The same as the previous entry, except that the "Define" for the letter (its VRAM position) is divided by two, for use in the split-screen mode.
        • XOff
          		•  - The X offset from the TitleCard's position at which this letter should be placed.

        The "Defines" used for these entries are listed at the top of the file.

        To effectively insert a space "character", simply use the "XOff" entry to place the character after the "space" farther away from the one before.
